# 持久化方案

## 数据结构（方案 B：每个宠物独立文档）

### 集合：`user_prefs`

用户偏好，一个用户一个文档。

```json
// user_prefs/{openId}
{
  "_openId": "oXXXX",
  "activePetId": "danta",
  "unlockedPets": ["xiaotuanzi", "danjuan", "maolizi", "danta"],
  "createdAt": "2026-05-20T...",
  "updatedAt": "2026-05-21T..."
}
```

### 集合：`pet_states`

每个用户的每个宠物独立一个文档。

```json
// pet_states/{openId}_{petId}
{
  "_openId": "oXXXX",
  "_petId": "danta",
  "energy": 72,
  "affection": 65,
  "mood": "开心",
  "currentScene": "idle",
  "sleepLoopCount": 0,
  "sleepTargetLoops": 0,
  "lastInteractionAt": 1716278400000,
  "consecutiveDays": 3,
  "lastActiveDate": "2026-05-20",
  "updatedAt": 1716278400000
}
```

## 宠物类型

| 类型 | 说明 | 可见性 | 解锁方式 |
|------|------|--------|---------|
| 基础款 | 免费大众宠物 | 所有用户可见 | 默认解锁 |
| 收费款 | 付费宠物 | 所有用户可见 | 付费解锁 |
| 私有定制款 | 客户专属定制 | 仅该客户可见 | 定制时绑定，其他用户不可见不可购买 |

### 用户可选宠物的判断逻辑

```
用户能看到的宠物 = 
  type === 'free' 的全部
  + type === 'paid' 的全部（未解锁的显示锁 + 价格）
  + type === 'private' 且 ownerOpenId === 当前用户openId 的
```

## 启动流程

```
1. 读取 user_prefs/{openId}
   - 不存在 → 创建默认（activePetId = bootstrap.defaultPetId，unlockedPets = 所有 free 宠物）
   - 存在 → 拿到 activePetId

2. 读取 pet_states/{openId}_{activePetId}
   - 不存在 → 创建默认状态
   - 存在 → 恢复状态，补算离线期间的衰减

3. 加载该宠物的 manifest → 开始播放
```

## 切换宠物流程

```
1. 用户在 petPicker 选择新宠物
2. 保存当前宠物状态 → pet_states/{openId}_{oldPetId}
3. 更新 user_prefs → activePetId = newPetId
4. 读取 pet_states/{openId}_{newPetId}（不存在则创建默认）
5. 加载新宠物 manifest → 重新开始播放
```

## 保存时机

| 时机 | 保存内容 |
|------|---------|
| 每 30 秒 tick（防抖 10 秒） | pet_states |
| 页面 hide / 小程序切后台 | pet_states |
| 切换宠物时 | pet_states（旧宠物）+ user_prefs |
| AI 回复后 | pet_states（心情变了） |

## 离线衰减

用户关闭小程序后再打开时：

```
elapsed = (now - state.updatedAt) / 1000
state = tick(state, elapsed)  // 补算离线期间的精力衰减
```

上限：最多补算 24 小时（避免离开一周回来精力直接归零）

## 用户记忆系统（轻量自研）

### 核心理念

记忆属于用户，不属于宠物。所有宠物共享同一份用户记忆。用户跟 A 宠物说过的话，B 宠物也知道。这让用户感觉"这些宠物都懂我"。

### 三类信息架构

```
┌─────────────────────────────────────────────────┐
│  对话记录（5轮）                                  │
│  存储：小程序内存（不持久化）                      │
│  作用：当前会话上下文连贯                         │
│  清空：关闭小程序 / 超过5分钟没说话               │
└─────────────────────────────────────────────────┘
                    +
┌─────────────────────────────────────────────────┐
│  短期记忆（8 条，每条 ≤20字）                     │
│  存储：云端 user_memories 集合                    │
│  作用：跨会话的"最近发生的事"                     │
│  写入：AI 输出 memory 时立即写入                  │
│  淘汰：满 8 条时淘汰综合分最低的                  │
└─────────────────────────────────────────────────┘
                    +
┌─────────────────────────────────────────────────┐
│  用户画像（1 段文本，100-150字）                   │
│  存储：云端 user_profiles 集合                    │
│  作用：稳定的用户认知                             │
│  更新：每积累 3 条新记忆触发 AI 重新生成           │
└─────────────────────────────────────────────────┘
```

### 数据存储

#### 集合：`user_memories`（短期层，8 条上限）

```json
// user_memories/{自动ID}
{
  "_openId": "oXXXX",
  "content": "用户喜欢吃榴莲",
  "importance": 0.8,
  "createdAt": "2026-05-20T14:30:00Z"
}
```

- 不做实体聚合，不覆盖，每条独立存储
- AI 输出 memory 时直接追加
- 满 8 条时淘汰综合分最低的

#### 集合：`user_profiles`（长期层，每用户一个文档）

```json
// user_profiles/{openId}
{
  "_openId": "oXXXX",
  "portrait": "互联网公司上班，经常加班。养了一只猫叫咪咪，最近不爱吃饭。喜欢吃榴莲，不喜欢下雨天。性格偏内向，喜欢安静的陪伴。周末喜欢爬山。",
  "memoryCountSinceUpdate": 2,
  "lastUpdatedAt": "2026-05-19T10:00:00Z"
}
```

### 短期记忆淘汰机制

**综合分 = importance + 时间加成**

```
时间加成 = max(0, 1 - 已过去的小时数 / 72)
```

| 时间 | 时间加成 | importance=0.8 的综合分 | importance=0.4 的综合分 |
|------|---------|----------------------|----------------------|
| 刚写入 | 1.0 | 1.8 | 1.4 |
| 6小时前 | 0.92 | 1.72 | 1.32 |
| 24小时前 | 0.67 | 1.47 | 1.07 |
| 48小时前 | 0.33 | 1.13 | 0.73 |
| 72小时+ | 0 | 0.8 | 0.4 |

淘汰规则：满 8 条时，计算所有记忆的综合分，删除最低分的那条。

效果：
- 新记忆有 72 小时的"新鲜保护"，不会被立即淘汰
- 72 小时后时间加成归零，纯靠 importance 决定去留
- 高 importance 的记忆（用户核心偏好/事实）长期留存，最终被画像吸收

### 记忆写入规则

- 不做实体聚合，不覆盖
- AI 自行判断是否值得记住，输出 memory 字段（content + importance）
- importance 范围 0-1，由 AI 打分
- 每条记忆独立存储，满了淘汰综合分最低的
- 重复信息的合并交给画像更新来处理

### 主回复 AI（每次对话）

**输入（messages 格式）：**
```
[system] 角色 + 规则 + 画像 + 短期记忆 + 宠物状态    ~300 tokens
[user] 第1轮用户说的话                               \
[assistant] 第1轮AI回复                               |
[user] 第2轮用户说的话                                | 5轮历史 ~250 tokens
[assistant] 第2轮AI回复                               |
...                                                  /
[user] 当前用户输入                                   ~30 tokens
```

**输出：**
```json
{
  "reply": "辛苦啦~ 快休息吧",
  "emotion": "gentle",
  "nextAction": "sleep-enter",
  "memory": {
    "content": "用户今天又加班到很晚",
    "importance": 0.6
  }
}
```

- `memory` 是可选字段，AI 自行判断是否值得记住
- `importance` 由 AI 打分（0-1），越重要越高
- 大部分闲聊不会输出 memory

### 画像更新 AI（低频触发）

**触发条件：** 每积累 3 条新记忆触发一次。

**输入：**
```
当前画像 + 全部 8 条记忆
```

**输出：** 新的用户画像文本（100-150字），合并去重后覆盖旧画像。

### Token 成本总结

| 环节 | 频率 | 每次成本 | 平均每次对话分摊 |
|------|------|---------|----------------|
| 主回复（画像+记忆+5轮历史） | 每次对话 | ~630 tokens | 630 tokens |
| 记忆写入 | ~30% 的对话 | 0（合并在主回复里） | 0 |
| 画像更新 | 每 3 条新记忆 | ~350 tokens | ~35 tokens |
| **总计** | | | **~665 tokens/次对话** |

### 给 AI 的完整信息结构

```javascript
{
  // 用户说了什么
  userText: "今天加班到十点，好累啊",
  
  // 宠物身份
  petId: "danta",
  petName: "蛋挞",
  
  // 宠物当前状态
  petState: {
    energy: 35,
    affection: 72,
    mood: "温柔",
    timeOfDay: "晚上"
  },
  
  // 关系等级
  relationship: "连续3天",
  
  // 长期层：用户画像
  portrait: "互联网公司上班，经常加班。养了一只猫叫咪咪...",
  
  // 短期层：近期记忆
  memories: [
    "用户昨天说周末想去爬山",
    "咪咪最近不爱吃饭",
    "用户不喜欢下雨天",
    ...
  ],
  
  // 对话记录（5轮）
  chatHistory: [
    { user: "我今天很累", pet: "辛苦啦~" },
    { user: "是啊加班到十点", pet: "这么晚！快休息吧" },
    ...
  ]
}
```
