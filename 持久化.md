# 持久化方案

## 数据结构（方案 B：每个宠物独立文档）

### 集合：`user_prefs`

用户偏好，一个用户一个文档。

```json
// user_prefs/{openId}
{
  "_openId": "oXXXX",
  "activePetId": "danta",
  "unlockedPets": ["xiaotuanzi", "danjuan", "maolizi", "danta"],
  "createdAt": "2026-05-20T...",
  "updatedAt": "2026-05-21T..."
}
```

### 集合：`pet_states`

每个用户的每个宠物独立一个文档。

```json
// pet_states/{openId}_{petId}
{
  "_openId": "oXXXX",
  "_petId": "danta",
  "energy": 72,
  "affection": 65,
  "mood": "开心",
  "currentScene": "idle",
  "sleepLoopCount": 0,
  "sleepTargetLoops": 0,
  "lastInteractionAt": 1716278400000,
  "consecutiveDays": 3,
  "lastActiveDate": "2026-05-20",
  "updatedAt": 1716278400000
}
```

## 宠物类型

| 类型 | 说明 | 可见性 | 解锁方式 |
|------|------|--------|---------|
| 基础款 | 免费大众宠物 | 所有用户可见 | 默认解锁 |
| 收费款 | 付费宠物 | 所有用户可见 | 付费解锁 |
| 私有定制款 | 客户专属定制 | 仅该客户可见 | 定制时绑定，其他用户不可见不可购买 |

### 宠物配置扩展（bootstrap config 或 pets 集合）

```json
{
  "id": "danta",
  "name": "蛋挞",
  "type": "free",
  "price": 0,
  "ownerOpenId": null
}

{
  "id": "premium-cat-01",
  "name": "布丁",
  "type": "paid",
  "price": 1990,
  "ownerOpenId": null
}

{
  "id": "custom-dog-zhangsan",
  "name": "旺财",
  "type": "private",
  "price": 0,
  "ownerOpenId": "oXXXX_zhangsan"
}
```

### 用户可选宠物的判断逻辑

```
用户能看到的宠物 = 
  type === 'free' 的全部
  + type === 'paid' 的全部（未解锁的显示锁 + 价格）
  + type === 'private' 且 ownerOpenId === 当前用户openId 的
```

## 启动流程

```
1. 读取 user_prefs/{openId}
   - 不存在 → 创建默认（activePetId = bootstrap.defaultPetId，unlockedPets = 所有 free 宠物）
   - 存在 → 拿到 activePetId

2. 读取 pet_states/{openId}_{activePetId}
   - 不存在 → 创建默认状态
   - 存在 → 恢复状态，补算离线期间的衰减

3. 加载该宠物的 manifest → 开始播放
```

## 切换宠物流程

```
1. 用户在 petPicker 选择新宠物
2. 保存当前宠物状态 → pet_states/{openId}_{oldPetId}
3. 更新 user_prefs → activePetId = newPetId
4. 读取 pet_states/{openId}_{newPetId}（不存在则创建默认）
5. 加载新宠物 manifest → 重新开始播放
```

## 保存时机

| 时机 | 保存内容 |
|------|---------|
| 每 30 秒 tick（防抖 10 秒） | pet_states |
| 页面 hide / 小程序切后台 | pet_states |
| 切换宠物时 | pet_states（旧宠物）+ user_prefs |
| AI 回复后 | pet_states（心情变了） |

## 离线衰减

用户关闭小程序后再打开时：

```
elapsed = (now - state.updatedAt) / 1000
state = tick(state, elapsed)  // 补算离线期间的精力衰减
```

上限：最多补算 24 小时（避免离开一周回来精力直接归零）

## 用户记忆系统（轻量自研）

### 核心理念

记忆属于用户，不属于宠物。所有宠物共享同一份用户记忆。用户跟 A 宠物说过的话，B 宠物也知道。这让用户感觉"这些宠物都懂我"。

### 双层记忆架构

```
┌─────────────────────────────────────────┐
│  长期层：用户画像（1 段文本，100-150字）    │
│  稳定的用户认知，低频更新                  │
│  "互联网公司上班，养猫叫咪咪，喜欢榴莲..." │
└─────────────────────────────────────────┘
                    +
┌─────────────────────────────────────────┐
│  短期层：近期记忆（15 条，每条 ≤20字）     │
│  最近对话中提取的具体信息，高频写入         │
│  "用户今天加班很累"                       │
│  "咪咪最近不爱吃饭"                      │
│  "用户下周想去爬山"                       │
└─────────────────────────────────────────┘
```

AI 每次对话拿到的 = 用户画像 + 15 条记忆，短期和长期都齐全。

### 数据存储

#### 集合：`user_memories`（短期层，15 条上限）

```json
// user_memories/{自动ID}
{
  "_openId": "oXXXX",
  "content": "用户的猫咪咪最近不爱吃饭",
  "entity": "咪咪",
  "importance": 0.7,
  "createdAt": "2026-05-20T14:30:00Z"
}
```

#### 集合：`user_profiles`（长期层，每用户一个文档）

```json
// user_profiles/{openId}
{
  "_openId": "oXXXX",
  "portrait": "互联网公司上班，经常加班。养了一只猫叫咪咪，最近不爱吃饭。喜欢吃榴莲，不喜欢下雨天。性格偏内向，喜欢安静的陪伴。周末喜欢爬山。",
  "memoryCountSinceUpdate": 3,
  "lastUpdatedAt": "2026-05-19T10:00:00Z"
}
```

### 主回复 AI（每次对话）

**输入：**
```
system prompt（角色 + 规则）
+ 用户画像（~50-70 tokens）
+ 15 条记忆（~150 tokens）
+ 宠物状态（~30 tokens）
+ 用户原文（~20-50 tokens）
```

**输出：**
```json
{
  "reply": "辛苦啦~ 快休息吧",
  "emotion": "gentle",
  "nextAction": "sleep-enter",
  "memory": {
    "content": "用户今天又加班到很晚",
    "entity": "工作"
  }
}
```

- `memory` 是可选字段，AI 自行判断这句话是否值得记住
- 大部分闲聊（"嗯嗯"、"哈哈"、"你好"）不会输出 memory
- `entity` 用于实体聚合（同 entity 的记忆会被合并）

### 记忆写入逻辑（云函数执行）

```
AI 返回了 memory 字段？
  │
  ├─ 否 → 不写记忆，结束
  │
  └─ 是 → 检查 entity
         │
         ├─ 已有同 entity 的记忆 → 用新 content 覆盖旧的（实体聚合）
         │
         └─ 没有同 entity → 新增一条
                │
                └─ 超过 15 条？→ 删除 importance 最低的一条
```

### 重要性评分

写入时计算：

| 条件 | 分数 |
|------|------|
| 基础分 | 0.4 |
| 包含偏好表达（喜欢/讨厌/最爱） | +0.2 |
| 包含强烈情绪（难过/生气/想哭/幸福） | +0.2 |
| 包含事实信息（名字/年龄/工作/家人） | +0.15 |
| 用户原文较长（>30字） | +0.05 |

时间衰减：每天 -0.01，最低 0.1。

### 画像更新 AI（低频触发）

**触发条件：** 每积累 5 条新记忆触发一次。

**输入：**
```
当前画像 + 全部 15 条记忆
```

**Prompt：**
```
根据以下用户记忆，更新用户画像（100-150字）。
要求：
- 合并同一实体的信息（如多次提到同一个人/宠物/地点）
- 去除已过时的信息（如"明天要开会"已经过了）
- 保留最新状态
- 语言简洁，像一份人物档案

当前画像：
{旧画像}

近期记忆：
{15条记忆列表}

输出新画像（100-150字）：
```

**输出：** 一段新的用户画像文本，直接覆盖 `user_profiles/{openId}` 的 `portrait` 字段。

**成本：** ~300-400 tokens/次，每积累 5 条新记忆才触发一次，频率很低。

### Token 成本总结

| 环节 | 频率 | 每次成本 | 平均每次对话分摊 |
|------|------|---------|----------------|
| 主回复（含记忆召回） | 每次对话 | ~350 tokens | 350 tokens |
| 记忆写入 | ~30% 的对话 | 0（合并在主回复里） | 0 |
| 画像更新 | 每 5 条新记忆 + 周一 | ~350 tokens | ~20 tokens |
| **总计** | | | **~370 tokens/次对话** |

### 给 AI 的完整信息结构

```javascript
{
  // 用户说了什么
  userText: "今天加班到十点，好累啊",
  
  // 宠物身份
  petId: "danta",
  petName: "蛋挞",
  
  // 宠物当前状态
  petState: {
    energy: 35,
    affection: 72,
    mood: "温柔",
    timeOfDay: "晚上"
  },
  
  // 关系等级
  relationship: "连续3天",
  
  // 长期层：用户画像
  portrait: "互联网公司上班，经常加班。养了一只猫叫咪咪...",
  
  // 短期层：近期记忆
  memories: [
    "用户昨天说周末想去爬山",
    "咪咪最近不爱吃饭",
    "用户不喜欢下雨天",
    ...
  ]
}
```
