# 后台管理系统开发计划

## 1. 建设目标

后台管理系统用于把当前依赖代码和脚本维护的运营配置，逐步迁移到可视化、可审计、可发布的管理台。第一阶段聚焦前台小程序展示控制：

- 控制前台展示哪些宠物。
- 控制前台展示哪些背景房间。
- 控制每个宠物的展示名、副标题、预览图、透明视频素材。
- 控制默认宠物、默认背景和首页基础文案。
- 支持配置预览、草稿保存、正式发布和回滚。

后台不直接替代小程序渲染逻辑，而是管理 CloudBase 数据库中的 `app_configs/bootstrap`、`pets/{petId}` 和后续拆分出的素材配置。前台继续通过 `bootstrap`、`getPetManifest` 云函数读取配置。

## 2. 当前项目现状

### 2.1 前台读取链路

当前首页启动配置来源分三层：

1. `miniprogram/config/bootstrap.ts`：小程序本地 fallback。
2. `cloudfunctions/bootstrap/bootstrap.default.json`：云函数内置 fallback。
3. `app_configs/bootstrap.config`：CloudBase 数据库线上配置。

前台实际启动时调用 `bootstrap` 云函数。云函数优先读取数据库，读取不到时使用内置默认配置。

### 2.2 宠物素材链路

当前主宠物透明视频运行时统一使用双通道 H.264 MP4：

- 左半边：RGB。
- 右半边：alpha mask。
- 小程序用 `wx.createVideoDecoder()` 解码，再用 WebGL shader 合成透明画面。

每个宠物在 `bootstrap.pets` 中至少需要：

- `id`
- `name`
- `subtitle`
- `manifestKey`
- `videoUrl`
- `thumbUrl`
- `listenFrameUrl`
- `enabled`

每个宠物还可以在 `pets/{petId}` 中保存完整 manifest，用于动作、素材基础路径和性格配置。

### 2.3 背景房间链路

当前背景房间在 `bootstrap.rooms` 中维护，支持：

- 静态图片背景。
- 云端 H.264 MP4 视频背景。

每个房间至少需要：

- `id`
- `name`
- `subtitle`
- `kind`
- `mediaUrl`
- `thumbUrl`
- `enabled`

## 3. 管理后台范围

### 3.1 第一阶段范围

第一阶段只做运营配置后台，不做复杂素材加工：

- 宠物列表管理。
- 宠物展示开关。
- 宠物名称、副标题编辑。
- 宠物默认排序和默认宠物设置。
- 宠物视频、预览图、倾听图 URL 维护。
- 背景房间列表管理。
- 背景展示开关。
- 背景名称、副标题、类型、媒体 URL 维护。
- 默认背景设置。
- 首页文案和基础配置管理。
- 后台上传 WebM alpha 源素材。
- 后台自动验收 alpha：`ffprobe`、`alphaextract`、右半 alpha mask 检查。
- 后台自动转双通道 H.264 MP4。
- 后台自动抽取宠物预览帧。
- 后台自动上传 MP4 和预览帧到云存储。
- 后台自动生成宠物 manifest。
- 保存草稿、发布线上、查看当前线上配置。

素材自动化第一版就进入后台闭环。现有脚本和交付规范继续作为底层能力与验收标准，但入口从手工命令迁移到后台页面。

### 3.2 第二阶段范围

第二阶段增强素材治理和发布能力：

- 素材处理任务队列。
- 失败重试和处理日志。
- 批量上传宠物素材。
- 多动作素材包：`idle`、`listening`、`reply`。
- 素材版本管理和旧素材回滚。
- 发布前预览小程序实际配置。
- 配置版本历史和回滚。

### 3.3 暂不纳入第一阶段

- 用户管理、会员、支付、广告位策略。
- AI 回复 prompt 管理。
- 多租户系统。
- 复杂素材在线剪辑。
- 完整内容审核流。

## 4. 明确技术栈

### 4.1 应用形态

在仓库根目录新增一个独立 Web 管理台目录：

```text
admin/
  package.json
  src/
  public/
  server/
```

第一版技术栈固定为：

- 前端框架：Vite + React + TypeScript。
- 前端路由：React Router。
- 数据请求：TanStack Query。
- 表单和校验：React Hook Form + Zod。
- 样式：普通 CSS + CSS Variables，先不引入重型 UI 组件库。
- 图标：lucide-react。
- 管理 API：Node.js + Express，放在 `admin/server/`。
- 云端配置读写：CloudBase Node SDK + `cloudfunctions/adminConfig` 双通道。第一版本地管理服务直接调用 CloudBase，云函数用于线上权限边界和后续部署。
- 素材处理：Node.js 调用系统 `ffmpeg` / `ffprobe`，复用现有转码、alpha 验收、上传规则。
- 云存储上传：沿用 `cos-nodejs-sdk-v5` 和 CloudBase Manager SDK。
- 构建工具：npm workspace 或独立 `admin/package.json`，不影响小程序构建。

原因：

- 和小程序代码隔离，避免影响前台构建。
- 本仓库可以统一维护数据结构、脚本和文档。
- 素材自动化需要稳定调用 `ffmpeg`，第一版放在 Node 管理服务中比放进云函数更稳。
- 未来管理台前端可以部署到 CloudBase 静态托管，管理服务部署到带 ffmpeg 的内部服务器。

### 4.2 后端接口形态

新增管理云函数，同时新增本地/服务器管理 API。前端不直接写数据库。

```text
cloudfunctions/adminConfig/
admin/server/
```

`adminConfig` 云函数核心接口：

- `getDraftConfig`
- `getPublishedConfig`
- `saveDraftConfig`
- `publishConfig`
- `rollbackConfig`
- `listPets`
- `upsertPet`
- `disablePet`
- `listRooms`
- `upsertRoom`
- `disableRoom`

`admin/server` 素材接口：

- `POST /api/media/pets/inspect`
- `POST /api/media/pets/transcode`
- `POST /api/media/pets/upload`
- `POST /api/media/pets/create-from-webm`

好处：

- 权限集中控制。
- 可以做字段校验。
- 可以写操作日志。
- 可以避免误写线上配置。
- 素材处理和配置发布分离，避免一个长耗时转码请求影响配置发布。

## 5. 数据模型规划

### 5.1 保留现有线上配置

继续保留：

```text
app_configs/bootstrap
pets/{petId}
```

`bootstrap` 是前台线上读取的最终配置。

### 5.2 新增草稿配置

新增集合：

```text
admin_config_drafts
```

建议文档：

```json
{
  "_id": "bootstrap",
  "config": {},
  "status": "draft",
  "updatedAt": "",
  "updatedBy": ""
}
```

后台编辑默认写入草稿，不直接影响前台。

### 5.3 新增发布版本

新增集合：

```text
admin_config_versions
```

每次发布写一条版本记录：

```json
{
  "_id": "auto",
  "version": "2026-05-18-001",
  "config": {},
  "publishedAt": "",
  "publishedBy": "",
  "summary": "新增毛栗子，删除布丁奶盖豆豆"
}
```

用于回滚和审计。

### 5.4 新增操作日志

新增集合：

```text
admin_audit_logs
```

记录：

- 操作人。
- 操作类型。
- 修改对象。
- 修改前后摘要。
- 时间。
- 请求来源。

## 6. 权限设计

第一阶段采用简单白名单：

- 在云函数环境变量中配置允许的管理员 openid 或邮箱。
- 管理云函数每次写操作先校验身份。
- 未登录或非白名单用户只能读取公开配置，不能保存和发布。

后续可升级为角色：

- `viewer`：只读。
- `editor`：可保存草稿。
- `publisher`：可发布线上。
- `admin`：可管理成员和回滚。

## 7. 页面规划

### 7.1 登录页

功能：

- 管理员登录。
- 显示当前环境 ID。
- 显示权限状态。

第一阶段可以用 CloudBase 登录态或临时管理口令，后续接入正式账号体系。

### 7.2 仪表盘

展示：

- 当前线上版本。
- 草稿是否有未发布改动。
- 当前启用宠物数量。
- 当前启用背景数量。
- 最近发布记录。

### 7.3 宠物管理

列表字段：

- 启用状态。
- 预览图。
- 名称。
- 副标题。
- 宠物 ID。
- 视频 URL。
- 排序。
- 最近更新时间。

编辑字段：

- `id`：创建后不建议修改。
- `name`：前台展示名。
- `subtitle`：前台选择页副标题。
- `enabled`：是否展示。
- `videoUrl`：双通道 MP4 云文件 ID。
- `thumbUrl`：选择页预览图。
- `listenFrameUrl`：倾听态预览图。
- `manifestKey`：manifest 标识。
- `frameOffset`：排序或旧预览偏移字段，后续可替换为 `sortOrder`。

操作：

- 新增宠物。
- 编辑宠物。
- 启用/隐藏宠物。
- 调整排序。
- 设置为默认宠物。
- 打开云文件地址。
- 上传 WebM alpha 源素材并自动创建运行时素材。
- 查看素材验收结果和转码结果。
- 校验必填字段。

### 7.4 背景管理

列表字段：

- 启用状态。
- 预览。
- 名称。
- 副标题。
- 类型：图片/视频。
- 媒体 URL。
- 排序。

编辑字段：

- `id`
- `name`
- `subtitle`
- `kind`
- `mediaUrl`
- `thumbUrl`
- `enabled`

操作：

- 新增背景。
- 编辑背景。
- 启用/隐藏背景。
- 设置默认背景。
- 校验图片/视频类型。

### 7.5 首页配置

可编辑：

- 应用名称。
- 首页提示文案。
- 默认宠物。
- 默认背景。
- 倾听光球视频 URL。
- 设置页会员小卡片文案。

### 7.6 发布中心

功能：

- 查看草稿和线上差异。
- 发布草稿到线上 `app_configs/bootstrap`。
- 写入版本记录。
- 回滚到指定版本。
- 发布前校验。

发布前校验：

- 默认宠物必须存在且启用。
- 默认背景必须存在且启用。
- 每个启用宠物必须有 `videoUrl` 和 `thumbUrl`。
- 每个启用背景必须有 `mediaUrl`。
- ID 不重复。
- URL 必须是 `cloud://`、`/pages/` 或允许域名。

### 7.7 素材自动化

第一版直接纳入后台。

功能：

- 上传宠物待机 WebM。
- 检查源素材是否是 VP9 WebM。
- 检查 `ALPHA_MODE=1`。
- 用 `libvpx-vp9` 抽取 alpha mask。
- 检查 alpha mask 的 `YMIN` 和 `YMAX`。
- 检查尺寸、帧率、时长、音频并给出通过/警告/失败结论。
- 转成 `1440x960 / 24fps / H.264 / yuv420p / 无音频` 的双通道 MP4。
- 裁右半边 mask 再检查一次。
- 抽取 `frame_0030.png` 和 `frame_0090.png`。
- 上传视频和预览帧到 `pets/{petId}/actions/idle/`。
- 自动回填宠物表单中的 `videoUrl`、`thumbUrl`、`listenFrameUrl`。
- 自动生成或更新 `pets/{petId}` manifest 草稿。

处理状态：

- `pending`：待处理。
- `inspecting`：验收中。
- `failed`：素材不合格或处理失败。
- `transcoding`：转码中。
- `uploading`：上传中。
- `ready`：素材已准备好，可发布。

第一版限制：

- 只处理单个 `idle` WebM。
- 只支持本机或部署机已安装 `ffmpeg` / `ffprobe` 的环境。
- 暂不做批量上传。
- 暂不做 `listening`、`reply` 多动作自动编排。

## 8. 配置生成规则

后台内部可以拆分管理宠物、背景和首页设置，但发布时生成前台仍然能直接消费的 `BootstrapConfig`：

```json
{
  "schemaVersion": 1,
  "configVersion": "admin-published-xxxx",
  "appName": "",
  "defaultPetId": "",
  "defaultRoomId": "",
  "defaultPetName": "",
  "homeHint": "",
  "homeMedia": {},
  "settings": {},
  "rooms": [],
  "pets": []
}
```

短期不改前台读取结构，减少风险。等后台稳定后，再考虑把 pets、rooms 拆成独立集合并由 `bootstrap` 云函数聚合。

## 10. 风险与约束

- 线上配置目前会被 `npm run seed:cloudbase -- --apply` 覆盖，后台上线后需要把 seed 脚本改成只初始化或需要显式确认覆盖。
- 管理后台必须有权限控制，不能直接暴露写数据库能力。
- 前台依赖 `bootstrap.pets` 和 `bootstrap.rooms`，发布配置时必须保持字段兼容。
- 宠物视频必须继续遵守双通道 MP4 规范，不能让普通 MP4 被配置为主宠物透明视频。
- 删除宠物应先隐藏，确认无用户数据依赖后再做物理删除。
- 云存储素材不应在后台误删，第一阶段只从前台列表隐藏。

## 11. 第一版验收标准

- 管理台可以看到当前线上宠物和背景。
- 可以隐藏或展示某个宠物。
- 可以修改宠物名称，并发布到前台。
- 可以隐藏或展示某个背景。
- 可以设置默认宠物和默认背景。
- 可以上传单个宠物待机 WebM。
- 可以自动完成素材验收、双通道 MP4 转码、预览帧抽取和云存储上传。
- 可以自动把处理后的 `videoUrl`、`thumbUrl`、`listenFrameUrl` 回填到宠物配置。
- 可以自动生成 `pets/{petId}` manifest 草稿。
- 发布后小程序重新打开能看到新配置。
- 发布记录可查询。
- 至少能回滚到上一个版本。

## 12. 近期执行建议

1. 先把当前配置整理为后台初始线上数据，只保留小团子、蛋卷、毛栗子。
2. 把 `seed-cloudbase-data.js` 增加保护，避免未来误覆盖后台发布内容。
3. 新增 `adminConfig` 云函数，实现读线上配置、保存草稿、发布和回滚。
4. 新建 `admin/` 管理台，实现宠物管理、背景管理、发布中心。
5. 同步实现 `admin/server` 素材自动化接口，把上传 WebM 到生成宠物配置的流程纳入第一版。
6. 第一版完成后，再补任务队列、失败重试和多动作素材包。
