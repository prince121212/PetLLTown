# 视频提示词和调度策略

本文档用于统一“云养宠”主宠物视频素材的生成、验收、命名、manifest 接入和运行时调度策略。整体采用“双基准帧锚定模型”：所有动作要么从同一个状态锚点出发并回到同一个锚点，要么负责连接两个状态锚点。目标是在不打断当前播放视频的前提下，让宠物具备更真实、更随机、更有情绪反馈的生命感。

## 1. 总体原则

### 1.1 双基准帧

每只宠物必须先确定两张核心锚点图：

| 锚点 | 文件建议 | 姿态 | 作用 |
| --- | --- | --- | --- |
| 清醒基准帧 A | `{petId}-anchor-awake.png` | 自然站立或坐姿，面向镜头，脚底稳定 | 清醒状态下所有待机、微动作、互动反馈的首尾对齐基准 |
| 睡眠基准帧 B | `{petId}-anchor-sleep.png` | 趴下或蜷缩闭眼，姿态稳定 | 睡眠状态下所有循环、睡眠微动作的首尾对齐基准 |

要求：

- 同一只宠物的所有素材必须共享同一套画面坐标、主体比例、脚底高度和视觉安全区。
- 清醒基准帧 A 和睡眠基准帧 B 本身必须是透明 PNG，尺寸为 `720x960`。
- 所有视频生成时都要以对应锚点作为首帧、尾帧约束。
- 对齐目标是“用户视觉无跳帧”，不是只在 prompt 中口头声明。素材进入项目之前必须做首尾帧抽检。

### 1.2 视频分类

| 类型 | 首尾帧规则 | 用途 |
| --- | --- | --- |
| 清醒锚点视频 | 首帧 = 尾帧 = 清醒基准帧 A | `awake-*`，包含待机、微动作、倾听、回应、触摸反馈等所有清醒态动作 |
| 睡眠锚点视频 | 首帧 = 尾帧 = 睡眠基准帧 B | `sleep-*`，包含睡眠循环和睡眠微动作 |
| 跨状态转场视频 | 首帧 = 前状态锚点，尾帧 = 后状态锚点 | `transition-awake-to-sleep`、`transition-sleep-to-awake`，连接清醒和睡眠 |

命名规则必须和锚点规则一致：

- 所有 A -> A 视频统一使用 `awake-` 前缀。
- 所有 B -> B 视频统一使用 `sleep-` 前缀。
- 所有 A -> B 或 B -> A 视频统一使用 `transition-` 前缀。
- 不保留旧的 `idle`、`reply`、`micro-*` 等通用运行时命名，调度、manifest、云存储路径都直接使用本套 actionId。

### 1.3 项目素材规格

本项目首页主宠物运行时统一使用双通道 H.264 MP4，不直接播放 alpha WebM。

源素材首选：

```text
格式：WebM
编码：VP9
透明：真实 alpha 通道
尺寸：720x960
帧率：24fps
音频：无
背景：透明背景
主体：完整入镜，位置、大小、脚底高度一致
```

运行时素材：

```text
格式：MP4 / H.264 / yuv420p
尺寸：1440x960
帧率：24fps
音频：无
布局：左右双通道
左半：720x960 RGB
右半：720x960 alpha mask
```

注意：

- 不使用绿幕作为最终交付素材。绿幕只可作为中间生成手段，进入项目之前必须抠成真实 alpha。
- 视频不内嵌音频。叫声、呼吸声、抚摸声等如需使用，应作为独立 `audioUrl` 资源配置。
- 每条动作视频都必须可独立验收、独立上传、独立配置到 manifest。

## 3. 统一提示词约束

以下提示词中的 `{宠物描述}`、`{petId}`、`帧 A`、`帧 B` 需要按具体宠物替换。

所有提示词都必须附加以下通用要求：

```text
通用技术要求：
以我上传的宠物参考图和锚点帧为准，保持角色外观、毛色、体型、五官、服饰或特征完全一致。
输出 720x960、24fps、透明背景、真实 alpha 通道、主体完整入镜、固定镜头、无文字、无水印、无阴影贴图、无环境背景。
宠物位置稳定，主体大小稳定，脚底或身体接地点与对应锚点帧完全一致。
视频不要内嵌音频。
首帧和尾帧必须严格对齐指定锚点帧，保证动作结束后可以无缝接入下一条视频。
```

如果生成工具不能直接输出真实 alpha，可先输出干净纯色背景版本，但交付项目之前必须重新抠成真实 alpha 源视频，并按项目验收标准检查。

## 6. 跨状态转场视频提示词

跨状态视频是唯一首尾锚点不同的素材。

### 6.1 `transition-awake-to-sleep`

```text
根据我上传的宠物参考照片、清醒基准帧 A 和睡眠基准帧 B，生成一条 4 秒的入睡过渡视频。

动作要求：
宠物从清醒基准姿态 A 开始，慢慢放松身体，逐渐趴下或蜷缩，眼睛缓缓闭合，耳朵自然放平，尾巴收拢贴近身体，最后稳定过渡到睡眠基准姿态 B。整个过程缓慢、自然、舒适，像是真的困了要睡觉。不要突然倒下，不要夸张打哈欠，不要离开原位。首帧必须与清醒基准帧 A 一致，尾帧必须与睡眠基准帧 B 一致。
```

### 6.2 `transition-sleep-to-awake`

```text
根据我上传的宠物参考照片、睡眠基准帧 B 和清醒基准帧 A，生成一条 4 秒的唤醒过渡视频。

动作要求：
宠物从睡眠基准姿态 B 开始，像是被轻轻唤醒，先耳朵微动，然后缓缓睁眼，抬起头看向镜头，最后慢慢站起或坐起，恢复到清醒基准姿态 A。整个过程自然、慵懒、可爱，像刚睡醒还有点迷糊。不要突然弹起，不要夸张伸懒腰，不要离开原位。首帧必须与睡眠基准帧 B 一致，尾帧必须与清醒基准帧 A 一致。
```

## 7. 运行时调度策略

### 7.1 核心播放模型

运行时维持长度为 5 的播放队列：

```text
[当前播放] -> [下一个预加载] -> [候选 3] -> [候选 4] -> [候选 5]
```

原则：

- 当前播放的视频不被打断。
- 事件发生时保留队列头部，后续队列统一交给调度算法重新计算。
- 队列第二个视频应提前预加载，保证衔接更稳。
- 调度层只关心“当前锚点”和“目标意图”，不为每种事件手写固定队列。
- 清醒锚点 A 和睡眠锚点 B 不能直接互通，跨状态必须经过 `transition-awake-to-sleep` 或 `transition-sleep-to-awake`。
- 同一锚点内的视频可以任意互相衔接，具体选择由权重、冷却、数值和事件优先级共同决定。

### 7.2 调度输入状态

调度算法参考当前宠物状态，但不使用单一硬阈值直接决定视频。现有状态面板已经展示的数值包括：

| 状态 | 字段 | 用途 |
| --- | --- |
| 精力 | `energy` / `statEnergy` | 影响疲惫、活跃、入睡、打哈欠等视频权重 |
| 亲密度 | `affection` / `statAffection` | 影响开心回应、贴近感、触摸反馈等视频权重 |
| 心情 | `mood` / `moodText` | 影响开心、好奇、温柔、困了、兴奋等情绪动作权重 |
| 连续陪伴天数 | `consecutiveDays` / `relationshipText` | 影响关系展示，也可作为长期关系权重参考 |

内部调度还会用到：

- `currentScene`：当前播放场景。
- `sleepLoopCount`：已连续播放的睡眠循环次数。
- `sleepTargetLoops`：本轮睡眠目标循环次数。
- `lastInteractionAt`：最近一次互动时间。
- `lastActiveDate`、`updatedAt`：活跃日期和状态更新时间。

### 7.3 清醒随机微动作

清醒锚点 A 下，每 10-30 秒生成一个随机微动作倒计时。倒计时结束后，算法把“随机微动作”作为一个候选意图参与本轮调度。

候选规则：

| 状态 | 可选微动作 |
| --- | --- |
| 精力高 | `awake-scratch`、`awake-lick`、`awake-tilt`、`awake-tail`、`awake-look-around` |
| 精力正常 | `awake-scratch`、`awake-lick`、`awake-yawn`、`awake-tilt`、`awake-look-around` |
| 精力低 | `awake-yawn`、`awake-lick`、`awake-look-around` |
| 心情低落 | `awake-yawn`、`awake-look-around` |

微动作播放完后仍处于清醒锚点 A，下一条可以是任意 `awake-*` 视频。通常由算法根据当前数值回到某个 `awake-idle-*`，也可以继续接入更高优先级的清醒交互视频。

### 7.4 互动响应

用户语音交互：

```text
清醒锚点 A -> awake-listening -> awake-reply-* -> 清醒锚点 A
```

AI emotion 到视频映射：

| AI emotion | 回应视频 |
| --- | --- |
| `happy`、`excited`、`positive` | `awake-reply-happy` |
| `shy`、`comforted` | `awake-reply-shy` |
| `confused`、`question`、`unknown` | `awake-reply-confused` |
| `sad`、`sorry`、`low` | `awake-reply-sad` |
| 其他 | `awake-reply-normal` |

触摸交互：

```text
清醒锚点 A -> awake-touch-petting -> 清醒锚点 A
```

触摸可以同步更新亲密度，例如 `affection + 1`，但数值更新不应阻塞视频播放。

### 7.5 睡眠调度

进入睡眠：

```text
清醒锚点 A -> transition-awake-to-sleep -> 睡眠锚点 B
```

触发条件：

- `energy < 20`
- 用户明确说“睡觉”“休息一下”等睡眠意图
- 后续可增加夜间时间段权重

睡眠中：

- 基础循环播放 `sleep-loop`。
- 每 30-60 秒随机触发 `sleep-ear-twitch` 或 `sleep-tail-twitch`。
- 睡眠微动作播放完回到 `sleep-loop`。

退出睡眠：

```text
睡眠锚点 B -> transition-sleep-to-awake -> 清醒锚点 A
```

触发条件：

- 用户唤醒宠物或开始语音交互。
- 能量恢复到配置化阈值。
- 达到配置的最大睡眠时长。

睡眠中用户说话：

```text
睡眠锚点 B 当前播放不打断 -> transition-sleep-to-awake -> awake-listening -> awake-reply-* -> 清醒锚点 A
```

### 7.6 合法路径

合法路径只由锚点决定：

| 当前锚点 / 视频 | 允许下一个视频 |
| --- | --- |
| 任意 `awake-*` | 任意 `awake-*`，或在需要进入睡眠时接 `transition-awake-to-sleep` |
| `transition-awake-to-sleep` | 任意 `sleep-*` |
| 任意 `sleep-*` | 任意 `sleep-*`，或在需要回到清醒时接 `transition-sleep-to-awake` |
| `transition-sleep-to-awake` | 任意 `awake-*` |

禁止路径：

- `awake-*` 不能直接接 `sleep-*`。
- `sleep-*` 不能直接接 `awake-*`。
- `transition-awake-to-sleep` 不能接 `awake-*`。
- `transition-sleep-to-awake` 不能接 `sleep-*`。

### 7.7 调度算法

每次需要补充队列时，按以下步骤决定下一条视频：

1. 读取当前播放队列最后一个视频的 `anchorEnd`，确定当前位置是清醒锚点 A 还是睡眠锚点 B。
2. 汇总当前意图：用户语音、AI 回应、触摸、入睡、唤醒、随机微动作、自然待机。
3. 根据意图生成候选 actionId 列表。AI 返回的 `nextAction` 只能作为候选，不直接等于最终播放视频。
4. 如果目标意图属于另一个锚点，先解析对应转场视频；转场可用时播放转场，转场播放完成后再进入目标锚点的视频选择。
5. 如果目标意图和当前锚点一致，从该锚点的视频池中按权重选择候选视频。
6. 所有候选视频必须经过“素材可用性解析”。只有 manifest 中存在、启用、且至少有一个可播放 `videoUrls` 的 action 才能进入最终队列。
7. 如果第一候选素材缺失，按同锚点降级链查找替代视频。降级过程不能违反锚点合法路径。
8. 对刚播放过的视频设置冷却，避免短时间内重复出现同一微动作或同一回应。
9. 优先级相同的候选视频使用加权随机，权重由数值、时间间隔、交互上下文、素材可用性和最近播放历史共同决定。

清醒锚点 A 的优先级：

| 优先级 | 意图 | 候选视频 |
| --- | --- | --- |
| 100 | 用户正在说话 | `awake-listening` |
| 90 | AI 刚生成回复 | `awake-reply-*` |
| 80 | 用户触摸宠物 | `awake-touch-petting` |
| 70 | 需要进入睡眠 | `transition-awake-to-sleep` |
| 50 | 随机微动作到点 | `awake-scratch`、`awake-lick`、`awake-yawn`、`awake-tilt`、`awake-tail`、`awake-look-around` |
| 10 | 自然待机 | `awake-idle-normal`、`awake-idle-energetic`、`awake-idle-tired`、`awake-idle-sad` |

睡眠锚点 B 的优先级：

| 优先级 | 意图 | 候选视频 |
| --- | --- | --- |
| 100 | 用户唤醒或开始说话 | `transition-sleep-to-awake` |
| 80 | 能量恢复或睡眠结束 | `transition-sleep-to-awake` |
| 40 | 随机睡眠微动作到点 | `sleep-ear-twitch`、`sleep-tail-twitch` |
| 10 | 继续睡眠 | `sleep-loop` |

自然待机权重方向：

| 候选视频 | 主要加权因子 |
| --- | --- |
| `awake-idle-normal` | 默认基础权重最高，作为清醒态稳定底盘 |
| `awake-idle-energetic` | 精力较高、心情偏开心或兴奋、近期互动积极时提高权重 |
| `awake-idle-tired` | 精力下降、长时间未休息、接近入睡但尚未触发转场时提高权重 |
| `awake-idle-sad` | 心情偏低、互动减少、AI 情绪判断偏安慰或低落时提高权重 |

微动作选择权重：

- 每个微动作播放后进入冷却，建议 60-180 秒内不重复。
- `awake-yawn` 在低精力时提高权重。
- `awake-tail`、`awake-tilt` 在高心情或高亲密度时提高权重。
- `awake-look-around` 作为中性微动作，可在任何清醒状态下低权重出现。
- 睡眠微动作之间应交替出现，避免连续两次选择同一个 `sleep-*` 微动作。

### 7.8 素材可用性解析与降级

调度算法必须允许某只宠物暂时缺少部分视频素材。缺素材时，系统不报错、不空播、不跨锚点硬跳，而是在当前锚点内寻找语义最接近的可用视频。

一个 action 只有同时满足以下条件，才视为可播放：

- manifest 中存在该 action。
- action 没有被禁用。
- `anchorStart`、`anchorEnd` 与当前路径要求一致。
- `videoUrls` 至少有一个可用地址。
- 该地址已通过云文件解析或本地缓存检查。

清醒锚点 A 降级链：

| 目标意图 | 首选 | 降级顺序 |
| --- | --- | --- |
| 自然待机 | `awake-idle-*` | 同组其他 `awake-idle-*` -> 任意可用 `awake-*` 中最安静的视频 |
| 开心回应 | `awake-reply-happy` | `awake-reply-normal` -> `awake-listening` -> 可用 `awake-idle-*` |
| 害羞回应 | `awake-reply-shy` | `awake-reply-normal` -> `awake-listening` -> 可用 `awake-idle-*` |
| 疑惑回应 | `awake-reply-confused` | `awake-tilt` -> `awake-reply-normal` -> `awake-listening` -> 可用 `awake-idle-*` |
| 低落回应 | `awake-reply-sad` | `awake-idle-sad` -> `awake-reply-normal` -> 可用 `awake-idle-*` |
| 倾听 | `awake-listening` | `awake-tilt` -> `awake-look-around` -> 可用 `awake-idle-*` |
| 触摸 | `awake-touch-petting` | `awake-tail` -> `awake-reply-happy` -> `awake-reply-normal` -> 可用 `awake-idle-*` |
| 清醒微动作 | 对应 `awake-*` 微动作 | 同组其他清醒微动作 -> 可用 `awake-idle-*` |

睡眠锚点 B 降级链：

| 目标意图 | 首选 | 降级顺序 |
| --- | --- | --- |
| 继续睡眠 | `sleep-loop` | 任意可用 `sleep-*` |
| 睡眠微动作 | `sleep-ear-twitch` / `sleep-tail-twitch` | `sleep-loop` -> 任意可用 `sleep-*` |

跨状态转场降级规则：

- 如果 `transition-awake-to-sleep` 缺失，不能直接从 `awake-*` 切到 `sleep-*`。调度应继续留在清醒锚点 A，优先选择 `awake-idle-tired` 或其他可用 `awake-*`，并记录“缺少入睡转场素材”。
- 如果 `transition-sleep-to-awake` 缺失，不能直接从 `sleep-*` 切到 `awake-*`。调度应继续留在睡眠锚点 B，优先选择 `sleep-loop` 或其他可用 `sleep-*`，并记录“缺少唤醒转场素材”。
- 如果目标锚点内没有任何可用视频，该宠物素材包视为不可用，应回退到默认宠物或阻止切换到该宠物。

最终选择流程：

```text
目标意图 -> 候选 actionId 列表 -> manifest 可用性过滤 -> 同锚点降级 -> 冷却过滤 -> 加权随机 -> 最终 actionId
```

例如 AI 希望播放 `awake-idle-sad`，但该素材缺失：

```text
awake-idle-sad 缺失
-> 查找其他 awake-idle-*：awake-idle-normal 可用
-> 播放 awake-idle-normal
-> 记录缺失素材，后续后台可提示补齐 awake-idle-sad
```

## 8. Manifest 建议

### 8.1 动作配置

manifest 中每个视频都应作为独立 action 配置，示例：

```json
{
  "id": "awake-reply-happy",
  "type": "anchor",
  "label": "开心回应",
  "fps": 24,
  "anchorStart": "awake",
  "anchorEnd": "awake",
  "durationMs": 4000,
  "tags": ["awake", "reply", "happy"],
  "weight": 1,
  "videoUrls": [
    "cloud://.../pets/xiaotuanzi/actions/awake-reply-happy/videos/xiaotuanzi-awake-reply-happy-alpha-pack-h.mp4"
  ]
}
```

跨状态转场视频示例：

```json
{
  "id": "transition-awake-to-sleep",
  "type": "transition",
  "label": "入睡过渡",
  "fps": 24,
  "anchorStart": "awake",
  "anchorEnd": "sleep",
  "durationMs": 4000,
  "tags": ["transition", "sleep"],
  "videoUrls": [
    "cloud://.../pets/xiaotuanzi/actions/transition-awake-to-sleep/videos/xiaotuanzi-transition-awake-to-sleep-alpha-pack-h.mp4"
  ]
}
```

调度层以 `anchorStart`、`anchorEnd`、`tags`、`weight` 为核心字段计算路径和权重，不再依赖每个 action 手写 `next` 列表。

### 8.2 命名规则

源透明视频：

```text
{petId}-{actionId}-transparent-src.webm
```

运行时双通道 MP4：

```text
{petId}-{actionId}-alpha-pack-h.mp4
```

云存储路径：

```text
pets/{petId}/actions/{actionId}/videos/{petId}-{actionId}-alpha-pack-h.mp4
pets/{petId}/actions/{actionId}/anchors/{petId}-anchor-awake.png
pets/{petId}/actions/{actionId}/anchors/{petId}-anchor-sleep.png
```

## 9. 最终目标

这套体系的核心不是单纯增加视频数量，而是让所有视频都围绕两个稳定锚点组织起来：

- 清醒状态下，用户看到的是同一个宠物在不同心情和互动下自然变化。
- 睡眠状态下，宠物不是静态摆件，而是有轻微生命体征和偶发微动作。
- 交互发生时，当前视频不被粗暴打断，但下一段会迅速响应用户。
- 素材越多，随机调度越自然；锚点越稳定，视频越多也不会把系统变乱。

只要锚点、透明素材、manifest 和队列调度四件事保持一致，22 条视频可以直接构成现阶段完整的宠物生命感基础。
