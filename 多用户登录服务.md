# 多用户登录服务

## 目标

把小程序从纯游客体验升级为“游客可先看、登录后可同步”的多用户系统。

核心要求：
- 首页默认可进入，保持游客态
- 用户进入设置页时，引导登录
- 登录采用微信官方标准小程序登录流程
- 后端、云数据、管理后台都能识别同一个用户身份
- 现有按 `openId` 存储的用户数据继续沿用，不推翻现有结构

## 现状

当前项目已经有这些基础：
- `pet_states`：按用户和宠物保存状态
- `user_prefs`：用户偏好
- `user_memories`：短期记忆
- `user_profiles`：用户画像
- `voice_logs`：语音识别日志
- `ai_logs`：AI 回复日志

这些数据天然就是多用户结构，问题主要在于：
- 前端没有明确的登录态
- 没有统一的“已登录 / 未登录”判断
- 登录后的身份状态没有专门落库
- 管理后台虽然能看数据，但还没有一个正式的用户账号视图

## 登录原则

1. 采用微信官方标准小程序登录
- 前端调用 `wx.login`
- 后端用 `code2Session` 换取用户身份
- 使用 `openid` 作为基础用户标识

2. 游客态和登录态分离
- 游客可以正常进入首页和查看默认宠物
- 设置页、同步数据、个性化能力需要登录

3. 身份统一
- 小程序内所有用户相关数据都归到同一个 `openId`
- 后续如需打通公众号，再额外考虑 `unionid`

4. 不破坏现有数据结构
- 继续使用现有集合
- 只新增必要的登录态与身份映射数据

## 用户流程

### 1. 未登录进入
- 用户第一次打开小程序
- 默认进入首页
- 看到默认宠物
- 当前状态为游客态

### 2. 点击设置
- 进入设置页前先检查登录态
- 若未登录，弹出登录提示
- 用户确认后开始微信登录流程

### 3. 登录成功
- 获取 `openid`
- 建立本地登录态
- 回填用户资料
- 允许进入设置页和后续个性化能力

### 4. 后续访问
- 重新进入小程序时，优先检查本地登录态
- 后端再校验云端身份是否仍有效

## 前端规划

### 首页
- 默认不强制登录
- 保持原有宠物体验
- 顶部或设置入口显示当前登录状态

### 设置页
- 作为登录触发入口
- 未登录时弹窗引导登录
- 登录后展示用户相关设置和同步状态

### 登录弹窗
- 文案清楚说明登录后能做什么
- 仅在用户主动进入设置时触发
- 避免首页一打开就打断体验

### 本地状态
- 增加 `auth` 或类似状态模块
- 保存：
  - 是否登录
  - `openId`
  - 登录时间
  - 最后校验时间

## 后端规划

### 新增登录云函数
建议新增一个专门的登录云函数，例如：
- `authLogin`

职责：
- 接收 `wx.login` 返回的 `code`
- 调用微信官方 `code2Session`
- 返回用户身份信息
- 必要时初始化用户档案

### 用户身份存储
建议保持以下原则：
- `openId` 作为基础身份主键
- 用户相关集合继续按 `openId` 写入
- 登录成功后可在 `user_profiles` 或单独集合里补一条身份记录

### 后端校验
所有会写用户数据的云函数都应确保：
- 能拿到 `openid`
- 如果没有用户身份，直接返回未登录错误

现有已经按 `wxContext.OPENID` 运行的云函数可以继续沿用：
- `petState`
- `aiRespond`
- `voiceTranscribe`
- `updatePortrait`

## 数据同步规划

### 需要同步的用户数据
- `pet_states`
- `user_prefs`
- `user_memories`
- `user_profiles`
- `voice_logs`
- `ai_logs`

### 同步原则
- 用户登录后，后端自然写入自己的 `openId`
- 不做手动合并到别的用户
- 不做跨用户共享记忆

### 登录后的首次初始化
可考虑初始化：
- 默认 `user_prefs`
- 首次 `user_profiles`
- 登录时间字段

## 管理后台规划

### 用户表
后台需要一张真正的用户表，基于现有数据聚合：
- `openId`
- 当前宠物
- 记忆条数
- 画像状态
- 最近活跃时间
- 语音日志数量
- AI 日志数量

### 用户详情
点进某个用户后，可查看：
- `user_prefs`
- `pet_states`
- `user_memories`
- `user_profiles`
- `voice_logs`
- `ai_logs`

### 登录可见性
后台不负责用户登录，但要能看到：
- 哪些用户已产生数据
- 哪些用户最近活跃
- 哪些用户还没有画像

## 风险和注意点

1. 不要把“公众号关注”当成小程序登录本体
- 当前阶段只做小程序官方登录

2. 游客体验不能被打断
- 登录只在设置页触发
- 首页默认可用

3. 登录态要稳定
- 需要处理 session 失效
- 需要做重新登录兜底

4. 现有数据不能丢
- 所有改动都要兼容现有 `openId` 数据

## 实施顺序

1. 新增登录云函数
2. 前端接入 `wx.login`
3. 设置页接入登录弹窗
4. 增加本地登录态管理
5. 云函数补充登录校验
6. 管理后台补用户表与详情页
7. 测试游客态、登录态、重进态、失效态

## 当前实现

已完成的内容：
- 新增 `authLogin` 云函数，使用 `wx.login` 的 `code` 调用微信 `code2Session`
- 新增 `users` 集合，记录 `openId`、首次登录、最后登录、最后活跃、登录次数
- 小程序首页保持游客可用
- 点击设置入口时，如果未登录，会弹窗引导微信登录
- 登录成功后恢复该用户的宠物偏好和宠物状态
- 未登录时不保存用户偏好和宠物状态
- 后台云数据新增 `users` 集合
- 后台用户表聚合登录账号、宠物状态、记忆、画像、语音日志、AI 日志
- `petState`、`voiceTranscribe`、`aiRespond` 会刷新 `users.lastActiveAt`

需要部署的云函数：
- `authLogin`
- `petState`
- `voiceTranscribe`
- `aiRespond`

新增环境变量：
- `WECHAT_APP_ID`
- `WECHAT_APP_SECRET`

部署脚本：
- `npm run deploy:auth-login`

## 最终目标

用户打开小程序时可以先无感体验，真正需要保存个性化数据时再完成微信登录。

后台能明确看到“谁在用、用了什么宠物、留下了哪些记忆、最近是否活跃”，并且所有数据都能按同一个微信身份串起来。
