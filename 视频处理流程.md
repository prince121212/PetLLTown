# 视频处理流程

本文档记录首页主宠物视频从“透明素材”到“小程序透明播放”的完整路径。当前验证成功的方案不是 PNG 帧序列兜底，而是：

```text
带 alpha 的源素材 -> 双通道 H.264 MP4 -> wx.createVideoDecoder 解码 -> WebGL shader 合成
```

后续新素材的交付规格先看 `素材交付规范.md`。只有通过素材验收后，才进入本文档后续的转码、上传和接入流程。

## 1. 明确目标

首页需要播放一个带透明背景的宠物视频，并叠在动态/静态首页背景上。

最初的 PNG 帧序列方案虽然容易实现透明，但性能压力大：

- 多张 PNG 需要频繁加载和缓存。
- 内存占用高。
- 帧率和首屏加载都不稳定。

因此目标改成使用单个视频文件承载动画，同时保留透明背景效果。

## 2. 先确认原视频是否真的带透明通道

不是所有“看起来抠过图”的视频都真的有 alpha。需要先用 `ffprobe` 检测。

示例命令：

```bash
ffprobe -hide_banner -show_streams -show_format -print_format json "/path/to/source-video"
```

合格源素材通常满足其中一种：

- VP9 WebM，并且 metadata 里有 `ALPHA_MODE=1`。
- ProRes 4444 MOV。
- PNG 序列，单帧是 RGBA。
- 像素格式包含 `rgba`、`yuva`、`bgra`、`gbrap` 等 alpha 信息。

这次排查的结论：

- `HEVC（Alpha）版本.mp4` 实际是 `hevc + yuv420p`，没有可用 alpha，不合格。
- `抠图后的猫.mp4`、`抠图和的猫格式.mov` 等也是普通 `h264 + yuv420p`，不合格。
- `生成布偶猫视频_transparent_fixed.webm` 是 VP9 WebM，带 `ALPHA_MODE=1`，合格。

## 3. 从源素材抽取 alpha 做证据确认

对 VP9 WebM 需要显式使用 `libvpx-vp9` 解码，否则有些 ffmpeg 构建可能读不到 alpha。

示例命令：

```bash
ffmpeg -hide_banner -y \
  -c:v libvpx-vp9 \
  -i "/Users/huangchangwei/Downloads/生成布偶猫视频_transparent_fixed.webm" \
  -frames:v 1 \
  -vf alphaextract \
  /tmp/petllt-alpha-test/webm-alpha-check.png
```

然后检查 alpha 图是否有黑白变化：

```bash
ffmpeg -hide_banner \
  -i /tmp/petllt-alpha-test/webm-alpha-check.png \
  -vf signalstats,metadata=print:file=- \
  -frames:v 1 \
  -f null -
```

如果 `YMIN` 接近 `0`，`YMAX` 接近 `255`，说明透明遮罩里确实有背景透明和主体不透明的信息。

## 4. 转成小程序可播放的双通道 MP4

微信小程序不要直接依赖 HEVC alpha 或 WebM alpha。更稳的运行时格式是普通 H.264 MP4：

- 编码：H.264
- 像素格式：`yuv420p`
- 一个视频流
- 左半边放 RGB 画面
- 右半边放 alpha mask

当前成功生成命令：

```bash
/usr/local/bin/ffmpeg -y \
  -c:v libvpx-vp9 \
  -i "/Users/huangchangwei/Downloads/生成布偶猫视频_transparent_fixed.webm" \
  -filter_complex "[0:v]format=rgba,split=2[rgbsrc][alphasrc];[rgbsrc]format=rgb24[rgb];[alphasrc]alphaextract,format=gray,format=rgb24[alpha];[rgb][alpha]hstack=inputs=2,scale=1440:960:flags=lanczos,format=yuv420p[v]" \
  -map "[v]" \
  -an \
  -r 24 \
  -c:v libx264 \
  -preset medium \
  -crf 20 \
  -profile:v high \
  -level 4.1 \
  -movflags +faststart \
  /tmp/petllt-alpha-test/xiaotuanzi-idle-alpha-pack-h.mp4
```

生成后的文件规格：

```text
H.264 / yuv420p / 1440x960 / 24fps
左半边：720x960 RGB
右半边：720x960 alpha mask
```

## 5. 检查双通道 MP4 是否正确

先看整体信息：

```bash
ffprobe -hide_banner -show_streams -show_format -print_format json \
  /tmp/petllt-alpha-test/xiaotuanzi-idle-alpha-pack-h.mp4
```

再单独裁出右半边 alpha mask：

```bash
ffmpeg -hide_banner -y \
  -i /tmp/petllt-alpha-test/xiaotuanzi-idle-alpha-pack-h.mp4 \
  -frames:v 1 \
  -vf "crop=720:960:720:0,format=gray" \
  /tmp/petllt-alpha-test/packed-alpha-half-check.png
```

继续检查黑白范围：

```bash
ffmpeg -hide_banner \
  -i /tmp/petllt-alpha-test/packed-alpha-half-check.png \
  -vf signalstats,metadata=print:file=- \
  -frames:v 1 \
  -f null -
```

如果右半边 alpha mask 的 `YMIN=0`、`YMAX=255`，说明打包成功。

涉及位置：

- `miniprogram/config/bootstrap.ts`
- `cloudbase/configs/bootstrap.default.json`


## 8. 删除 PNG 帧序列兜底

为了避免误判，首页主舞台不再使用 PNG 帧序列作为兜底。

已删除/移除的内容：

- `miniprogram/components/alpha-video/*`
- 首页 `usingComponents` 里的 `alpha-video`
- 首页 WXML 中的旧组件路径
- bootstrap 里的 `frameSequence`
- 云函数 merge 逻辑里的 `frameSequence`

当前首页主舞台只保留：

```xml
<canvas id="petVideoCanvas" type="webgl" class="pet-canvas"></canvas>
```

## 9. 前端渲染流程

运行时在 `miniprogram/pages/index/index.ts` 里完成：

1. 用 `wx.cloud.downloadFile` 下载云端 MP4 到本地临时路径。
2. 用 `wx.createVideoDecoder()` 解码 H.264 MP4。
3. 在 `requestAnimationFrame` 中调用 `decoder.getFrameData()` 获取 RGBA 帧数据。
4. 把整帧上传成 WebGL texture。
5. shader 从左半边采 RGB，从右半边采 alpha mask。
6. 输出透明宠物画面。

关键 shader 思路：

```glsl
vec4 color = texture2D(u_texture, rgbUv);
vec4 mask = texture2D(u_texture, alphaUv);
float alpha = mask.r;
```

## 10. 解决黑色矩形问题

第一次视频能显示，但背景出现黑色矩形。这个现象说明：

- 视频 RGB 已经渲染成功。
- PNG 帧兜底没有生效。
- 问题在 WebGL canvas 的 alpha 透传或宿主合成。

修复方式是在创建 WebGL context 时显式请求透明：

```ts
canvas.getContext('webgl', {
  alpha: true,
  premultipliedAlpha: true,
  preserveDrawingBuffer: false,
})
```

后续首页背景已经改成独立视频层，宠物主体 canvas 只负责输出透明宠物，不再把静态壁纸合进宠物 canvas。这样背景视频和主体透明视频保持两条独立链路。

## 11. 验证是否真的走新方案

开发者工具控制台应看到这些日志：

```text
alpha video start requested
alpha video cloud source resolved
alpha video decoder started
alpha video first frame
alpha video mask samples
```

重点判断：

- `alpha video first frame` 的 `width` 应是 `1440`，`height` 应是 `960`。
- `byteLength` 应等于 `width * height * 4`。
- `split` 应显示左半 RGB、右半 alpha。
- `alpha video mask samples` 中应同时出现接近 `0` 和接近 `255` 的 mask 值。

如果截图里没有黑框，且上述日志存在，就说明成功走的是当前透明视频方案，不是 PNG 兜底。

## 12. 最终结论

成功路径是：

```text
选择真正带 alpha 的 WebM 源
-> 用 ffmpeg 抽 alpha 验证
-> 打包成左右双通道 H.264 MP4
-> 上传云存储并更新配置
-> 删除 PNG 帧兜底
-> 小程序用 VideoDecoder 解码
-> WebGL shader 左取 RGB、右取 alpha
-> WebGL canvas 透明叠在首页背景层上
-> 真机截图验证成功
```

这条方案兼顾了透明效果和性能：主动画只需要一个 MP4 文件，避免了 PNG 帧序列的大量图片加载和内存压力。
